## 1. Objetivo

Iniciar la fase **Dock Interactions** sobre el modelo Dock certificado en D52.

D53 introduce **exclusivamente infraestructura de interacción UI**, completamente separada de:

- `DockState` v1 (`activePanelIds` · `sizes`)
- Registry / Registration / Visibility / Layout / Slots / `DOCK_FEATURES`

Manteniendo:

- Compatibilidad total con API Freeze D51 + D52
- Principio **Zero UX** hasta certificación D53
- Extensión **únicamente aditiva** (nuevo context; sin reabrir `DockState`)

**D53.1 no implementa código funcional.**

### 1.1 Principios de D53

| Principio | Contrato |
|-----------|----------|
| Separación estricta | Dock State ≠ Dock Interaction State |
| Nesting one-way | `DockProvider` → `DockInteractionProvider` · nunca al revés |
| API Freeze hasta D60 | Firmas y shapes de sesión fijas; stubs permitidos |
| Zero UX hasta certificación | Sin CSS nuevo · sin animaciones · sin cambios visuales |
| Move-only cuando corresponda | No rediseñar shell · no migrar Analysis Inspector |
| Sin lógica científica | `docking/` no importa graphs / math / statistics / analysis / export |
| Documentación append-only | Sin reescribir historia D51/D52 |

### 1.2 Qué responde D53

| Fase | Pregunta |
|------|----------|
| D51 | ¿Cómo se construye un sistema Dock? |
| D52 | ¿Qué puede hacer un Dock? (modelo) |

## 3. Separación Dock State vs Interaction State

### 3.1 Dock State (D51/D52 — congelado)

**Ownership:** `DockProvider` / `DockContext` / `DockState`.

| Responsabilidad | Detalle |
|-----------------|---------|
| Visibility lógica | `activePanelIds` (qué paneles están “mostrados” en el modelo) |
| Tamaños de zona | `sizes` (modelo; no mutado por D53) |
| Registry | Query · registration · entries |
| Layout / slots | Definición estática / modelo |
| Feature flags | `DOCK_FEATURES` |

**No es responsabilidad de Dock State:** foco teclado, ventana activa UI, hover pointer, sesiones drag/resize.

### 3.2 Dock Interaction State (D53 — nuevo)

**Ownership:** `DockInteractionProvider` / `DockInteractionContext` / `DockInteractionState`.

| Responsabilidad | Detalle |
|-----------------|---------|
| Focus | `focusedDock` |
| Activation UI | `activeDock` (única ventana activa) |
| Hover pointer | `hoverDock` |
| Drag session | `draggingDock` + `dragSession` |
| Resize session | `resizingDock` + `resizeSession` |

**No es responsabilidad de Interaction State:** mutar `activePanelIds`, mutar layout final, snapping, persistencia, lógica científica.

### 3.3 Contrato de separación

```text
DockState          = MODEL (what exists / is visible / sized)
InteractionState   = UI SESSION (what is focused / active / hovered / being dragged)
NEVER MERGE SHAPES
NEVER REOPEN DockState v1 FOR INTERACTION FIELDS
```

## 4. Arquitectura de nesting

### 4.1 Árbol objetivo

```text
DockRoot
 ├── DockProvider
 │
 └── DockInteractionProvider
      │
      └── children (DockZone / DockPanel / ...)
```

`DockRoot` anida ambos providers. El provider de interacción vive **dentro** del árbol de `DockProvider`.

### 4.2 Dependencia unidireccional (gobernanza)

```text
DockProvider
        ↓
DockInteractionProvider
        ↓
children
```

| Regla | Contrato |
|-------|----------|
| Permitido | Hijos leen `useDockContext` y/o `useDockInteraction` |
| Permitido | Interaction lee el **context** (`useDockContext`) solo hacia abajo en el árbol |
| Prohibido | `DockInteractionProvider` importa o instancia `DockProvider` |
| Prohibido | `DockProvider` / `DockContext` importan módulos de interaction |
| Prohibido | Dependencias circulares |

```text
ONE-WAY ONLY
DockProvider → DockInteractionProvider
NEVER DockInteractionProvider → DockProvider (import/instantiate)
```

## 5. Interaction Model

### 5.1 Eventos congelados

| Evento | Rol |
|--------|-----|
| Focus | Establece foco de teclado / a11y |
| Blur | Limpia foco |
| Activate | Establece la **única** ventana activa |
| Deactivate | Limpia ventana activa |
| DragStart | Inicia sesión de arrastre |
| DragMove | Actualiza posición actual de sesión |
| DragEnd | Cierra sesión de arrastre |
| ResizeStart | Inicia sesión de redimensionado |
| ResizeMove | Actualiza posición actual de sesión |
| ResizeEnd | Cierra sesión de redimensionado |
| Float | Documento únicamente — **no implementado en D53** |
| Dock | Documento únicamente — **no implementado en D53** |
| Close | Documento únicamente — **no implementado en D53** |

### 5.2 Float / Dock / Close

Quedan en el Interaction Model **solo como eventos documentales** para D54+.

- No forman parte de la API pública implementada en D53.
- No autorizan ventanas flotantes, re-docking visual ni cierre de paneles en esta serie.

```text
FLOAT / DOCK / CLOSE = DOCUMENTED ONLY
IMPLEMENTATION = D54+
```

## 6. Interaction State (campos)

| Campo | Semántica congelada |
|-------|---------------------|
| `focusedDock` | `string \| null` — dock con foco de teclado / a11y |
| `activeDock` | `string \| null` — **única** ventana activa UI |
| `hoverDock` | `string \| null` — último dock bajo el pointer; `null` al salir |
| `draggingDock` | `string \| null` — id con drag en curso (espejo de `dragSession?.dockId`) |
| `resizingDock` | `string \| null` — id con resize en curso |
| `dragSession` | `DockDragSession \| null` — payload de sesión drag |
| `resizeSession` | `DockResizeSession \| null` — payload de sesión resize |

### 6.1 hover ≠ focus ≠ active

| Campo | Significa | No significa |
|-------|-----------|--------------|
| `hoverDock` | Último dock bajo el pointer | No es foco teclado · no es ventana activa |
| `focusedDock` | Foco a11y / teclado | No implica hover · no implica active necesariamente (salvo reglas de activate) |
| `activeDock` | Ventana activa única | No es hover · no es visibility (`activePanelIds`) |

**Por qué importa ahora:** en D54 aparecerán Drop Targets. `hoverDock` será la base para hit-testing de drop; confundirlo con `activeDock` o `focusedDock` genera deuda técnica.

### 6.2 Activation vs Visibility

| Concepto | Owner | Campo |
|----------|-------|-------|
| Visibility (modelo) | `DockState` | `activePanelIds` |
| Activation (UI) | Interaction State | `activeDock` |

Un panel puede ser visible en el modelo y no ser la ventana activa UI. D53 **no** redefine `activePanelIds`.

### 6.3 Invariante de activation

Como máximo **un** `activeDock` no-null. Activar otro dock desactiva el anterior. `activate(id)` implica focus sobre `id`.
