## 3. Reglas `governance.layout.*`

### 3.1 `layout.noReactState`

| Campo | Valor |
|-------|-------|
| **Regla** | El Layout Engine no puede poseer ni mutar React state (`useState`, stores React, etc.). |
| **Motivación** | El engine es un resolver puro de modelo. El estado UI pertenece a capas React fuera de `layout-engine/`. Mezclar state acoplaría el shell a React y rompería testabilidad/pureza. |

### 3.2 `layout.noHooks`

| Campo | Valor |
|-------|-------|
| **Regla** | Prohibido usar React hooks (`useEffect`, `useMemo`, `useCallback`, `useContext`, hooks custom, etc.) dentro de `layout-engine/`. |
| **Motivación** | Los hooks implican ciclo de vida React. D54 exige un engine usable como biblioteca pura (Tree → resolve → Tree). |

### 3.3 `layout.noDom`

| Campo | Valor |
|-------|-------|
| **Regla** | Prohibido acceder al DOM (`document`, `window` layout reads, `ref` DOM, medidores de layout browser) dentro de `layout-engine/`. |
| **Motivación** | El modelo es declarativo. Medición/DOM pertenece a capas de presentación futuras, no al foundation. |

### 3.4 `layout.treeImmutable`

| Campo | Valor |
|-------|-------|
| **Regla** | El `LayoutTree` de modelo se trata como inmutable; `resolve` produce un árbol normalizado sin mutación in-place del input. |
| **Motivación** | Inmutabilidad evita efectos colaterales, facilita reasoning y prepara persistencia/presets (D55/D56) sobre snapshots estables. |

### 3.5 `layout.allowedImportsOnly`

| Campo | Valor |
|-------|-------|
| **Regla** | `layout-engine/` solo puede importar módulos permitidos (tipos/helpers propios del package; stdlib TypeScript seguro). Sin imports de UI chrome, docking, inspector, sidebar, toolbar, page, tokens CSS como dependencia de render. |
| **Motivación** | Mantener el engine desacoplado. Valores numéricos del default tree pueden **espejar** tokens, pero no importar el sistema visual. |

### 3.6 `layout.apiFrozen`

| Campo | Valor |
|-------|-------|
| **Regla** | Los 9 símbolos del API Freeze no pueden romperse hasta D56 salvo approved architectural change. |
| **Motivación** | Estabilidad contractual para D54.2–D56 (consumption, persistence, presets) sin churn de tipos. |

**Símbolos:** `LayoutNode` · `LayoutTree` · `LayoutRegion` · `LayoutVisibility` · `LayoutConstraints` · `LayoutNodeType` · `LayoutNodeId` · `LayoutEngine` · `LayoutEngineProps`

Ver [`D54.1-layout-engine-api-freeze.md`](D54.1-layout-engine-api-freeze.md).

### 3.7 `layout.enginePure`

| Campo | Valor |
|-------|-------|
| **Regla** | `LayoutEngine` es puro: misma entrada → misma salida; sin I/O, sin timers, sin side effects. |
| **Motivación** | Certificabilidad y reutilización. El wiring (D54.3) y la persistencia (D55) pueden componer sobre una función pura. |

### 3.8 `layout.noScientificImports`

| Campo | Valor |
|-------|-------|
| **Regla** | Prohibido importar graphs / math / statistics / analysis / export / scientific / supabase / `page` desde `layout-engine/`. |
| **Motivación** | Misma isolation que docking: UX infrastructure ≠ dominio científico. |

### 3.9 `layout.barrelFrozen`

| Campo | Valor |
|-------|-------|
| **Regla** | El barrel `layout-engine/index.ts` exporta la superficie pública congelada; no elimina símbolos freeze; no expone internals prohibidos como API pública accidental. |
| **Motivación** | Control de superficie. Consumers futuros (D55) dependen de un barrel estable. |
