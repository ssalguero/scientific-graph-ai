## 5. Árbol conceptual (congelado)

El Layout Engine describe el **shell espacial** del Workspace mediante un árbol declarativo.

```text
Root
├── Sidebar
├── Workspace
│   └── Toolbar
├── Dock
│   └── Inspector
├── Floating Layers
└── Overlays
```

### 5.1 Semántica de nodos

| Nodo | Rol espacial |
|------|--------------|
| `Root` | Contenedor raíz del shell |
| `Sidebar` | Chrome izquierdo |
| `Workspace` | Región central de trabajo |
| `Toolbar` | Barra de herramientas (hijo espacial de Workspace) |
| `Dock` | Host shell del sistema Dock (región; **no** organiza tabs/slots internos) |
| `Inspector` | Panel docked tipificado bajo Dock en el árbol del shell |
| `Floating Layers` | Capas flotantes (modelo; sin comportamiento float en D54) |
| `Overlays` | Overlays / toasts / capas no ancladas al flujo principal |

### 5.2 Mapeo a regiones

| Nodo | `LayoutRegion` |
|------|----------------|
| Sidebar | `LEFT` |
| Workspace | `CENTER` |
| Toolbar | `TOP` (dentro del flujo CENTER) |
| Dock | `RIGHT` |
| Inspector | `RIGHT` (panel bajo Dock) |
| Floating Layers | `FLOATING` |
| Overlays | `FLOATING` |

Valores numéricos de tamaño en el default tree (D54.2) deberán **alinearse** a tokens actuales (p. ej. sidebar ~280, inspector/dock right ~320) como **datos de modelo**, no como clases CSS.

## 6. Layout Regions (congelado)

```text
LEFT
RIGHT
TOP
BOTTOM
CENTER
FLOATING
```

| Región | Uso previsto |
|--------|--------------|
| `LEFT` | Sidebar / chrome izquierdo |
| `RIGHT` | Dock / Inspector / chrome derecho |
| `TOP` | Toolbar / barras superiores dentro del flujo |
| `BOTTOM` | Reservado (bottom dock / panels futuros) |
| `CENTER` | Workspace principal |
| `FLOATING` | Floating layers · overlays |

---

## 7. Layout Visibility (congelado)

```text
VISIBLE
HIDDEN
COLLAPSED
```

| Valor | Semántica |
|-------|-----------|
| `VISIBLE` | Nodo presente y ocupando espacio según constraints |
| `HIDDEN` | Nodo no visible (fuera del flujo visual) |
| `COLLAPSED` | Nodo colapsado (espacio mínimo / chrome reducido) |

D54 no implementa UI de collapse/hide; solo el **modelo**.

---

## 8. Layout Constraints (congelado)

| Campo | Tipo conceptual | Semántica |
|-------|-----------------|-----------|
| `minWidth` | number \| undefined | Ancho mínimo |
| `maxWidth` | number \| undefined | Ancho máximo |
| `minHeight` | number \| undefined | Alto mínimo |
| `maxHeight` | number \| undefined | Alto máximo |
| `collapsed` | boolean | Preferencia/estado de colapso a nivel constraint |
| `locked` | boolean | Nodo no redimensionable / no movible por layout |

El Engine (D54.2) **valida** constraints; no aplica CSS.

---

## 9. LayoutNode (modelo conceptual)

Cada nodo del árbol posee:

| Campo | Rol |
|-------|-----|
| `id` | Identificador estable (`LayoutNodeId`) |
| `type` | Tipo de nodo (`LayoutNodeType`) |
| `parent` | Id del padre (o null en Root) |
| `children` | Lista de ids hijos |
| `visibility` | `LayoutVisibility` |
| `size` | Dimensiones de modelo (width/height según aplique) |
| `constraints` | `LayoutConstraints` |
| `region` | `LayoutRegion` |

Shapes exactas de TypeScript: [`D54.1-layout-engine-api-freeze.md`](D54.1-layout-engine-api-freeze.md).

## 10. Responsabilidades (congeladas)

### 10.1 Layout Engine

| Responsabilidad | Sí / No |
|-----------------|---------|
| Resolver / normalizar árbol | **Sí** |
| Obtener nodo / hijos | **Sí** |
| Validar regiones y constraints | **Sí** |
| Ser autoridad del shell espacial | **Sí** |
| React state / hooks / DOM / CSS | **No** |
| Persistir layout | **No** (D55) |
| Organizar tabs/slots Dock | **No** |

### 10.2 WorkspaceLayout (D54.3)

| Responsabilidad | Sí / No |
|-----------------|---------|
| Único consumidor del Engine en D54 | **Sí** |
| Construir / resolver default `LayoutTree` | **Sí** |
| Conservar slots de render actuales | **Sí** |
| Cambiar estilos / tokens | **No** |
| Exponer Context/hooks de layout | **No** |

### 10.3 page.tsx (D54.3)

| Responsabilidad | Sí / No |
|-----------------|---------|
| Orquestar contenido de dominio en slots | **Sí** |
| Ser autoridad del layout | **No** (deja de serlo) |
| Decidir árbol canónico del shell | **No** |

### 10.4 Dock / Sidebar / Inspector / Toolbar (D54)

| Responsabilidad | Sí / No |
|-----------------|---------|
| Permanecer funcionalmente iguales | **Sí** |
| Consumir `LayoutTree` directamente | **No** (D55) |
| Cambiar UX | **No** |

### 10.5 DockLayoutDefinition (coexistencia — Opción A)

| Capa | Gobierna |
|------|----------|
| **Layout Engine** | Regiones · árbol · constraints · visibilidad del **shell** |
| **DockLayoutDefinition** | Tabs · slots internos · organización interna del Dock · estado propio Dock |

```text
Layout Engine          ≠   DockLayoutDefinition
(shell espacial)           (interior Dock)
```

No se depreca nada en D54. En D55/D56 el Dock podrá leer su **región** desde el LayoutTree; su organización interna permanece en `DockLayoutDefinition`.

## 11. Decisiones oficiales (congeladas)

### 11.1 Wiring D54.3 — Opción C (mínimo)

```text
WIRING = C (MINIMAL)
WorkspaceLayout = único consumidor del Layout Engine en D54
page.tsx = orquestación only (deja de ser autoridad)
Dock / Sidebar / Inspector / Toolbar = NO consumen LayoutTree en D54
FULL CONSUMPTION = D55
```

**Justificación:** D54 es Foundation, no Consumption. Mantiene “move infrastructure first” (D47 / D51 / D53), minimiza riesgo de regresión y aísla la certificación.

### 11.2 DockLayoutDefinition — Opción A (coexistencia)

```text
DOCK_LAYOUT_DEFINITION = A (COEXIST)
NO DEPRECATION IN D54
SHELL = Layout Engine
DOCK INTERIOR = DockLayoutDefinition
```
